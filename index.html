Mobile-first campaign platform

Flame Stamps

A production roadmap for turning the visual prototype into a branded mobile experience, a campaign data platform and, optionally, a public participatory installation.

Overview

From a personal stamp to a measurable campaign experience.

Flame Stamps is a mobile web experience for Flame Tree Season. Visitors capture or upload an image, apply branded shapes, filters and color treatments, then generate a downloadable stamp.

The production work can scale in three levels: a functional core platform, a moderated collective gallery, and a public digital tree installation powered by approved user submissions.

Proposal options

Three production scopes

Proposal 01

Core Campaign Platform

The minimum production version. Users create, save and share personal Flame Stamps through a polished mobile-first experience.

  • Landing page and mobile creation flow
  • Desktop QR handoff to mobile
  • Camera capture or image upload
  • Shape, filter and color selection
  • Stamp generation, album, download and native share
  • User/session ID, image storage and basic admin dashboard
Technical focus
Frontend stamp generator, Rails backend, database, storage, session model and basic campaign metrics.
Estimated development
4 to 5 weeks

Proposal 02

Collective Gallery

Adds public participation. Users can submit their stamps to a campaign gallery, while the team reviews and approves content.

  • Everything in Core
  • Submit-to-gallery flow
  • Consent for public display
  • Pending, approved, rejected and private states
  • Moderation dashboard
  • Public gallery and approved content feed
Technical focus
Consent tracking, moderation queue, public/private visibility, approved media feed and gallery frontend.
Estimated development
6 to 8 weeks

Proposal 03

Tree Installation

Extends the gallery into a public display layer where approved stamps become part of a growing digital Flame Tree.

  • Everything in Core and Collective Gallery
  • Approved stamp feed for display mode
  • Basic crowdsourced tree for one target resolution
  • Display animation and fallback screen
  • Technical handoff documentation
  • Optional WebGL, particles and responsive display modes
Technical focus
Installation frontend, curated feed integration, display-specific layout, refresh logic and optional generative rendering.
Estimated development
8 to 10 weeks

Technical foundation

The backend turns the stamp into campaign value.

The visual experience creates the stamp. The backend stores the participation behind it: user/session, selected shape, filter, color, timestamp, image record and campaign source.

Frontend

Mobile-first interface for image capture, visual treatment, stamp generation, album, download and sharing.

Backend

Rails API or server-rendered platform with database, image storage, sessions, admin dashboard and data deletion tools.

Public layer

Moderated gallery or installation feed using only approved submissions when the project becomes publicly participatory.

Technical decisions to confirm

  • Domain: campaign domain or subdomain.
  • Hosting: production hosting for frontend and backend.
  • Storage: persistent image storage and thumbnails.
  • Sessions: anonymous device session or registered users.
  • Data: what user/contact data is collected and why.
  • Consent: privacy copy, public display consent and deletion path.
  • Moderation: required only when content becomes public.
  • CDN: recommended for image-heavy public traffic.

Recommended roadmap

Build the core first, then open participation.

The safest production path is progressive: launch the individual experience, add public participation after moderation is ready, then use approved content for a display or installation.

  1. Core 4–5 weeks: create, save and share Flame Stamps with backend data attached.
  2. Collective Gallery 6–8 weeks: publish approved submissions as a curated campaign archive.
  3. Tree Installation 8–10 weeks: turn approved stamps into a public digital tree.

Production requirements

What must be defined before build

The current prototype should be treated as a visual reference. A production build needs clear decisions around technology, hosting, content, data, consent and operational responsibility.

Mandatory for Core

Final assets, domain, hosting, mobile flow, image generation, backend, database, storage, admin access and QA across mobile browsers.

Mandatory for Public Use

Consent states, moderation, approved publishing flow, privacy copy, deletion mechanism and responsibility for reviewing content.
