Development proposal

Flame Stamps

Technical scope for developing the Flame Stamps experience in three scalable production levels: visual core, campaign data platform and collective data engine.

Production Structure

The project is divided into three deliverables. Each level adds a different technical layer: first the visual experience, then persistent campaign data, and finally collective participation through a managed dashboard and public output.

1. Core Visual MVP

A client-side mobile experience focused on creating and exporting Flame Stamps. This version validates the main interaction without requiring backend, user accounts, persistent storage or dashboard.

Includes

  • Landing or entry screen.
  • Mobile-first interface.
  • Camera capture or image upload.
  • Selection of brand shapes.
  • Selection of color treatment or filter.
  • Canvas-based image processing.
  • Stamp generation.
  • Downloadable image output.
  • Native share support when available.
  • Local album stored in the browser.

Technical Specs

  • Frontend-only build.
  • HTML, CSS and JavaScript.
  • Camera API or file upload input.
  • Canvas rendering for filter, shape and export.
  • LocalStorage or IndexedDB for temporary album data.
  • No server-side image persistence.
  • No user registration.
  • No moderation dashboard.

Limitations

  • Data can be lost if the user clears browser storage.
  • The campaign team does not receive user data or image records.
  • No public gallery or collective output.

Estimated development: 3–4 weeks.

2. Campaign Data Platform

A production version with backend, database, user/session management, mailing capabilities and persistent image storage. This level turns each stamp into a measurable campaign record.

Includes

  • Everything in Core Visual MVP.
  • Production domain and hosting.
  • Backend application.
  • Database for users, sessions and stamps.
  • Image upload and persistent storage.
  • User registration or lightweight account flow.
  • Email capture and mailing-ready user records.
  • Password management, if account login is required.
  • Personal album connected to user/session data.
  • Admin dashboard for reviewing records.
  • Basic data export for campaign reporting.

Technical Specs

  • Rails backend or equivalent production framework.
  • Relational database for users, stamps and events.
  • Image storage service for originals and generated outputs.
  • SSL-enabled production environment.
  • Authentication or anonymous device-session model.
  • Mailer configuration for transactional or campaign emails.
  • Admin access for data management.
  • Basic analytics events for stamp creation, download and share.

Data Stored

  • User or session ID.
  • Name and email, if required by the campaign.
  • Generated stamp image.
  • Selected shape, filter and color.
  • Creation date and time.
  • Download and share events.
  • Campaign source, QR or entry point.

Limitations

  • No public gallery unless scoped.
  • No public tree or installation output.
  • No advanced moderation unless public publishing is added.

Estimated development: 6–8 weeks.

3. Collective Data Engine

A collective participation layer where user-generated stamps can be submitted, filtered, moderated and used as data for a public gallery, tree visualization or installation output.

Includes

  • Everything in Campaign Data Platform.
  • Submit-to-collective flow.
  • Consent for public use.
  • Submission states: private, pending, approved, rejected.
  • Dashboard for filtering and managing submissions.
  • Moderation actions: approve, reject, hide, delete.
  • Public or internal gallery feed.
  • Approved data feed for visualization.
  • Basic collective tree or visual output.

Dashboard Functions

  • View all submitted stamps.
  • Filter by date, user, shape, color, status or source.
  • Approve or reject public submissions.
  • Delete images or user records if required.
  • Export selected data.
  • Manage which images feed the public output.

Technical Specs

  • Moderation queue.
  • Public/private publishing states.
  • Approved content API or feed.
  • Gallery or installation frontend.
  • Optional CDN for image-heavy traffic.
  • Optional AI-assisted moderation.
  • Optional WebGL or particle-based visualization.

Limitations

  • Hardware is not included by default.
  • Venue setup is not included by default.
  • On-site technical support must be scoped separately.
  • Advanced generative visualization requires additional production time.

Estimated development: 8–12 weeks.

Required Before Development

  • Final confirmation of scope.
  • Approved brand assets and visual guidelines.
  • Domain or subdomain decision.
  • Hosting and deployment requirements.
  • Storage requirements for generated images.
  • User model: local, anonymous session or registered account.
  • Email and mailing requirements.
  • Data retention and deletion rules.
  • Consent and privacy copy.
  • Moderation responsibility if content becomes public.
  • Definition of whether the collective output is gallery, tree or installation.

Recommended Path

  1. Start with Core Visual MVP to validate the interaction, visual output and mobile experience.
  2. Move into Campaign Data Platform if the campaign needs users, email capture, persistent albums, data and reporting.
  3. Extend into Collective Data Engine if the goal is to create a public gallery, moderated archive or installation.
